# v0.23 — Source-binding-aware role disambiguation

## Problem

v0.22 resolves role pairings by scoring DOM candidates on size, position, text, attributes, and library weight. When two candidates score within 0.05 of each other (or one side's cross-verify sits in the 0.5-0.65 band), the algorithm picks the higher score and moves on. That's a guess. The "review uncertain mapping" UX we considered (screenshots + user y/n) puts the burden on the user and only catches cases where the algorithm already knew it was uncertain.

We have a stronger signal available: the Liquid source for both themes is already on disk.

- Live theme source: `base-cache/sections/<section>.liquid` (pulled via Shopify CLI)
- Dev theme source: `sections/<section>.liquid` (workspace)

Liquid variable bindings are the canonical definition of what a role *is*. The DOM element that renders `{{ product.title }}` is the product title, full stop. No threshold tuning, no empirical priors.

v0.23 makes source-binding the primary disambiguation signal and demotes screenshots to a rare fallback.

## Design

### Step 2.0.5 — Build source-binding map (NEW, runs before Step 2.1 library merge)

For each side (live, dev):

1. Parse the section's Liquid file using `@shopify/liquid-html-parser` (real AST, not regex). Pin exact version in `package.json`; audit release notes per upgrade.
2. Walk the Liquid AST and the snippet chain transitively. **Semantics:**
   - **Depth cap: 10 hops** (Dawn's `main-product → product-form → buy-buttons → product-variant-picker → product-variant-options → ... → swatch-input` is 8 deep; cap 6 silently loses coverage).
   - **Branch traversal: depth-first, all branches.** Both `{% if %}` branches and multiple `{% render %}` calls in the same snippet all walked. Order: lexical order in source.
   - **Cycle detection:** `visited` set keyed on `(filename, render_args_hash)` — NOT filename alone. Snippet B rendered with different `with` scopes produces different AST semantics; dedup by filename loses coverage. Use stable-JSON hash of the `RenderMarkup.args` array.
   - **AST cache:** parsed ASTs cached by `(filepath, mtime)` in-process. Latency budget: 5s added to a 20-section/10-role intake-anchors run. CI harness measures + fails build if exceeded.
3. Walk AST for Liquid output tags: `{{ product.title }}`, `{{ product.price }}`, `{{ product.description }}`, `{{ variant.price }}`, `{{ product.featured_image }}`, `{{ section.settings.title }}`.
4. Walk AST for `RenderMarkup` nodes (`{% render 'NAME' %}`, `{% render 'NAME' with X as Y %}`, `{% render 'NAME', foo: bar %}`, `{% include 'NAME' %}`).
   - **Symbol table required.** Track variable rebinding across render boundaries. `{% render 'price', product: card_product %}` means inside `price.liquid`, `product` resolves to the caller's `card_product`. Without this, variable-flow claims are fiction.
   - Initial implementation: exact variable names only (`product.title` match). Deeper substitution (e.g., `card_product.title` → `product.title`) deferred to v0.24.
5. Walk AST for `{% liquid %}` tag bodies using Liquid-statement grammar (tags appear without `{% %}` wrappers). Recurse into body with a different visitor.
6. Walk AST for form elements with `action="/cart/add"`.
7. For each match, capture:
   - The enclosing HTML element (outermost tag containing the Liquid output)
   - Stable attributes: `id`, `class`, `data-section-id`, `data-block-id`, `data-shopify-editor-id`
   - The enclosing `{% schema %}` block ID if inside one
   - **Liquid citation:** `{ filepath, line, column }` — node exposes `.position.start` character offset; build an offset→line map once per file (split on `\n`, binary search).
   - The full snippet call chain (e.g. `product-template.liquid:42 → product-form.liquid:8 → buy-buttons.liquid:15`) for decision-report citations.
8. **Role-to-locus cardinality:** a role may map to multiple loci (e.g. `primary_atc` appears in a main form AND a sticky mobile form). Capture all, rank by call-chain depth (shallowest wins for DOM-locus resolution). Decision report lists all loci found.
9. Build a role-to-locus map keyed on role name via `intake-anchors/role-bindings.json`.

### Step 2.3.5 — Source-binding verdict (veto semantics, not a score nudge)

After base scoring and before aggregation, compute `source_binding_match` per candidate.

For each candidate per role:

1. Look up the role's Liquid locus from Step 2.0.5.
2. If the locus has a stable selector (section ID, block ID, data-shopify-editor-id, or a class pattern present in both Liquid source and rendered DOM), query the live DOM for elements descending from that locus.
3. Check if the candidate is within that subtree.
4. Emit `source_binding_match: true | false | "inconclusive"`.

Inconclusive means the Liquid template uses a binding that can't be statically mapped to a DOM subtree. Common causes:
- Client-side rendering (Hydrogen section, React component)
- Snippet renders whose output varies at runtime (variant-dependent)
- Missing stable attributes on the enclosing element

### Ranking adjustment — veto, not a score delta

Source-binding is canonical theme source, not an empirical prior. The v0.23 rewrite treats it accordingly: a determinate verdict is a **gate**, not a score nudge. There is no magic delta to tune.

**Three-tier ranking (applied after Stage 1 element-type scoring):**

1. **Confirmed tier** — `source_binding_match: true`. Candidate is in the role's Liquid-derived DOM subtree. Rank these first.
2. **Inconclusive tier** — `source_binding_match: "inconclusive"`. Static analysis couldn't verify (client-side rendered, interpolated selector, missing stable attrs). Rank these second, using the v0.22 DOM score. This tier is the only place screenshot fallback triggers.
3. **Rejected tier** — `source_binding_match: false`. Candidate is outside the role's locus. Remove from consideration entirely.

Within each tier, rank by the existing v0.22 score (Stage 1 element-type rules, `[-0.20, +0.15]`, unchanged). The winner is the top-ranked candidate in the highest non-empty tier.

**Escape hatches** (when role-bindings.json is wrong):
- `--ignore-source-binding <role>` — fall back to pure v0.22 DOM scoring for that role.
- `--no-source-binding` — global disable for the run.
- Project config `<role>: {"ignore_source_binding": true}` — persistent per-project opt-out.

**Why veto instead of weighted scoring:** the plan's premise is "Liquid variable bindings are the canonical definition of what a role *is*." A DOM candidate outside the role's Liquid-declared subtree is wrong by construction, not just low-scoring. Scoring-based tuning (`+0.20 / −0.35` bounds) couldn't overcome a 0.21 base-score gap in the canonical wrong-pairing case reviewers flagged; veto semantics eliminate the gap entirely. Tuning treadmills are a code smell — when the signal is canonical, treat it as canonical.

**Worked example:**
- Wrong candidate: base 0.74, element pass, source fail → **rejected**. Removed from ranking.
- Correct candidate: base 0.53, element fail, source pass → **confirmed tier**. Wins by default (only confirmed candidate).
- If multiple candidates are confirmed: rank by v0.22 score. If one is in confirmed tier and another is inconclusive: confirmed wins regardless of score delta.

**Load-bearing guardrail:** the wrong-pairing regression suite (20 v0.22-verified-correct pairings) must still pass under veto semantics. A misconfigured binding that vetoes a correct candidate will be caught there before ship.

### Step 2.6 — Cross-verify integration

New signal: `source_binding_parity`. Emitted per role after both sides (live + dev) have run through Step 2.3.5's tier assignment.

- **Both sides confirmed** (both winners in their respective confirmed tiers) → `source_binding_parity: "both_confirmed"`. No cross-verify adjustment needed; this is the happy path.
- **One side confirmed, other inconclusive** → `source_binding_parity: "partial"`. Warning in decision report ("dev confirmed via Liquid source; live inconclusive"). No automatic rejection.
- **One side confirmed, other side's winner would have been rejected** → `source_binding_parity: "mismatch"`. **Dedicated warning** in decision report: strongest wrong-pairing signal we have. Pairing is flagged for manual review but not auto-rejected (escape hatch respected).
- **Both sides inconclusive** → `source_binding_parity: "both_inconclusive"`. Fall through to v0.22 DOM-based cross-verify; trigger screenshot fallback if the v0.22 uncertainty band applies.

Parity is a warning layer, not a hard block. This is intentional: `role-bindings.json` misconfiguration on one side (e.g., live pulled from a different snapshot than dev) must not silently drop pairings. The decision report surfaces the conflict; the user decides.

### Fallback: screenshots + user confirm

Triggered only when source-binding is inconclusive on both sides AND at least one of:
- Winner score in `[0.4, 0.55]`
- Cross-verify in `[0.5, 0.65]`
- Runner-up within 0.05 of winner

**Review batching** — screenshot prompts do NOT interleave with scoring. All roles are scored first, uncertain roles collected into a review queue, then prompted in a single dedicated review phase at the end of the run.

**Interactive mode (default, TTY detected):**
1. During Step 2.3 scoring, for any role matching the fallback trigger, capture `element.screenshot()` for the winning candidate on both sides (deferred; queued for review phase, not blocking).
2. Write to `.theme-forge/screenshots/<role>-live.png` and `<role>-dev.png`.
3. Write a static HTML file at `.theme-forge/screenshots/<role>-compare.html`:
   ```html
   <html><body>
   <h2>ROLE: product_title</h2>
   <div style="display:flex;gap:20px">
     <div><h3>LIVE (.product-title-wrapper h1)</h3>
       <img src="product_title-live.png"><p>Dainty Chain Necklace</p></div>
     <div><h3>DEV (.product-information h1)</h3>
       <img src="product_title-dev.png"><p>Dainty Chain Necklace</p></div>
   </div></body></html>
   ```
4. At end of run, print a dedicated review phase (batched, not interleaved with scoring):
   ```
   ──── Review uncertain mappings (2 roles) ────

   [1/2] subtitle
     source binding: inconclusive on both sides
     dev winner:  .product-information .subtitle (score 0.52)
     live winner: .product-single__subtitle (score 0.48)
     Screenshots: file:///Users/carl/project/.theme-forge/screenshots/subtitle-compare.html
     → Opening in default browser...
     Confirm pairing? [y]es / [n]o / [s]kip / [i]gnore-permanently:
   ```
5. User responses:
   - `y` — accept pairing, mark `source_binding_match: true` (user-override).
   - `n` — reject pairing, mark as `rejected`, remove from anchor-map. User must re-run with `--ignore-source-binding <role>` or edit `role-bindings.json` to correct.
   - `s` — skip (don't persist decision), same as non-interactive auto-skip.
   - `i` — ignore permanently, write `<role>: {"ignore_source_binding": true}` to project config so future runs skip source-binding for this role.

**Non-interactive mode (`CI=true`, `--no-prompt`, or no TTY):**
- Never hang. Auto-choose `skip` for every uncertain role.
- Write to `.theme-forge/anchors/<section>.decision-report.json → uncertain_roles[]` with screenshot paths.
- Exit code 0 (warnings, not errors).
- **End-of-run summary line on stdout (not stderr) so CI logs capture it:**
  ```
  WARN: 2 uncertain roles require review: [subtitle, reviews_summary]
        See .theme-forge/screenshots/ for visual comparison.
        Run interactively or set --fail-on-uncertain to block CI.
  ```
- `--fail-on-uncertain` flag (opt-in) exits 1 if any uncertain roles remain. Recommended for CI.

**Timeout:** interactive prompt times out after 60s per role → falls back to `skip` behavior, continues to next role.

No server, no build step. Opens in default browser via OS `file://` handler (macOS `open`, Linux `xdg-open`, Windows `start`).

### Step 6.5 — Decision report additions

Per role:
```json
"source_binding": {
  "live": {
    "locus_selector": "#shopify-section-product-template [data-block-id='title']",
    "tier": "confirmed",
    "winner_in_locus": true,
    "rejected_candidates": []
  },
  "dev": {
    "locus_selector": "#MainProduct-title",
    "tier": "confirmed",
    "winner_in_locus": true,
    "rejected_candidates": ["h2.related-title"]
  },
  "parity": "both_confirmed"
}
```

`tier` ∈ {`"confirmed"`, `"inconclusive"`, `"rejected"`} — matches Step 2.3.5's three-tier ranking. `rejected_candidates` lists any candidates that would have competed for the role but were vetoed by source-binding; recorded for diagnostic purposes (helps debug `--ignore-source-binding <role>` escape-hatch decisions).

Top-level summary:
```json
"source_binding_coverage": {
  "roles_confirmed": 10,
  "roles_inconclusive": 3,
  "roles_all_rejected": 1,
  "parity_mismatches": 1,
  "candidates_vetoed": 4
}
```

`roles_all_rejected` surfaces the failure mode where every candidate was rejected — usually means role-bindings.json is wrong or the theme doesn't have this role at all. User action: inspect, then either fix the binding or run with `--ignore-source-binding <role>`.

## Success criteria

- **≥20% reduction in cross-verify failures vs v0.22** on the benchmark matrix (not just one site)
- **Zero false auto-accepts** on the wrong-pairing regression suite (per Codex: failure reduction alone isn't a win if confident-wrong goes up)
- Screenshot fallback triggered **≤ once per 5 roles** on a typical product-information section (otherwise the inconclusive detector is too loose; tighten the Liquid→DOM selector resolver)
- Regression harness passes 3 existing + 4 new scenarios + multi-hop snippet chain fixture
- Per-role decision report includes readable Liquid file:line citation for every source-binding verdict (not a hidden score)

## Open questions

1. **How do we resolve the Liquid locus to a DOM selector?** Shopify server-rendered themes emit `data-section-id`, `data-block-id`, `data-shopify-editor-id`. Most themes have them; some custom ones strip. Need to verify coverage on the second-site benchmark before committing to attribute-based resolution.
2. **What's the graceful degradation when a theme strips all stable attributes?** Fall back to class-pattern matching (Liquid source has `class="product-title"`, look for `.product-title` in DOM). Less precise but still useful.
3. **Do we surface `wrong_pairings_flagged` as a hard block or a warning?** One-side-passes-other-fails is high-signal but not 100%. Propose: warning in decision report, not a block. User can override with `--ignore-source-binding-warnings`.
4. **What happens when the Liquid template uses `{% if %}` conditional rendering?** The single-hop parser captures both branches. If the DOM winner is in either, `source_binding_match: true`. Good enough for MVP; revisit if false-positives appear on the second-site eval.

## Debug affordances

- **`--trace-source-binding <role>`** — prints the full walk for that role: binding entry used, each snippet file visited (with call chain), matched variables/snippets/forms at each hop, final locus, locus→DOM resolution steps, rejected candidates. Writes to stdout with a clear header; does not alter scoring.
- **`--dump-liquid-ast <section>`** — prints the parsed Liquid AST for a section (post-cache lookup) in indented JSON for operator inspection when verdicts look wrong.
- **`--no-source-binding`** — global kill-switch; skips Step 2.0.5 and 2.3.5 entirely, falling back to v0.22 scoring. Useful for isolating source-binding regressions.
- **`--ignore-source-binding <role>`** (existing; promoted from error-registry buried reference) — per-role suppression. Documented in `--help`.
- **Citations are clickable.** Output format uses `file://` URLs where supported (macOS Terminal, iTerm2, VS Code integrated terminal): `product-template.liquid:42` renders as `file:///Users/carl/.../product-template.liquid:42`. Plain paths on dumb terminals.

## Upgrade migration

`/theme-forge upgrade --to v0.23` does:

1. **Pre-check** — if `.theme-forge/role-libraries/projects/<slug>.json` has custom role bindings already (via hand-edit or v0.22 `--update-project`), detect and preserve. No overwrite.
2. **Create `intake-anchors/role-bindings.json`** — only if absent. If present, leave alone (user-customized).
3. **Backfill decision reports** — for each v0.22 anchor map in `.theme-forge/anchors/`, add a `source_binding: { status: "not_available", reason: "pre-v0.23 anchor map; re-run intake-anchors to populate" }` stub. `--why` renders this stub as a note, not an error.
4. **Backup** — all files the migration writes are captured in `.theme-forge/.migrations/v0.23-backup.json` (existing framework).
5. **Rollback** — `/theme-forge upgrade --rollback v0.23` restores from backup.

### Dream state delta (before / after)

**Before (v0.22):** User runs `/theme-forge intake-anchors product-information-main`. Two candidates score 0.72 vs 0.71. Algorithm picks 0.72. Turns out 0.72 is the wishlist button, 0.71 is the ATC. Refine-section then fails silently — the "ATC" transformation is applied to the wishlist button. User discovers on visual QA, 30 minutes wasted.

**After (v0.23):** Same section. Same two candidates. Before aggregation, scorer checks: does candidate descend from the Liquid block that renders the role's binding (form `action="/cart/add"` inside the `product-form` snippet chain)? 0.72 is outside that subtree → −0.20 penalty → final 0.52. 0.71 is inside → +0.15 → final 0.86. Runner-up flipped to winner. Decision report shows: `primary_atc: winner=button.atc-btn, source_binding=match, citation=product-form.liquid:12 → buy-buttons.liquid:8`. User reads citation and trusts the answer. Zero minutes wasted.

### Error & Rescue Registry (known failure modes + recovery)

| Failure | Signal | Rescue |
|---|---|---|
| `@shopify/liquid-html-parser` throws on malformed Liquid | Parser exception with line number | Catch, log as `source_binding: "parse_error"`, fall through to pure DOM scoring (v0.22 behavior). Record in decision report. |
| Snippet chain exceeds 10 hops | Depth cap hit during recursion | Emit `source_binding: "depth_exceeded"` + cite the last resolved file. Fall through. |
| Snippet cycle (A renders B renders A) | Cycle detected in `visited` set | Emit `source_binding: "cycle_detected"`, cite both files. Fall through. |
| Theme strips all stable attributes (`data-section-id` etc) | Locus resolver produces no candidate subtree | Try class-pattern fallback from Liquid source. If still inconclusive, mark `inconclusive` and fall through. |
| Client-side rendered role (Hydrogen) | Liquid source has no output tag for role, but DOM has candidate | `inconclusive` on both sides → trigger screenshot fallback if score is in the uncertainty band |
| Role binding misconfigured (`role-bindings.json` wrong) | Source binding consistently fails on verified-correct DOM winners | User runs `--ignore-source-binding <role>` or edits `role-bindings.json`. Decision report prints binding used so error is diagnosable. |
| `base-cache/sections/<section>.liquid` missing | Live side Liquid unavailable | Source-binding runs on dev side only. Emit `source_binding_parity: "dev_only"`. Cross-verify falls back to v0.22 DOM-only behavior. |

### Failure Modes Registry (catastrophic scenarios + mitigations)

| Scenario | Worst outcome | Mitigation |
|---|---|---|
| v0.23 flips a currently-correct v0.22 pairing to wrong due to source-binding false negative | Production regression, user loses trust | Wrong-pairing regression suite: 20 pairings known correct under v0.22 must still be correct under v0.23. Ship only if zero regressions. |
| `@shopify/liquid-html-parser` supply-chain compromise | Parser executes hostile code at build time | Pin exact version in package.json, audit release notes per upgrade, sandbox parser calls (no filesystem/network access required by parser). |
| `role-bindings.json` becomes a maintenance tax (adding sections requires guessing variables for every theme family) | v0.24+ stalls, user forks the file | Ship tooling in v0.23: `/theme-forge suggest-bindings <role> <section>` scans existing Liquid + DOM + scorer output and proposes a binding entry. Reduces authoring cost to review-only. |
| Source-binding parity becomes a hard block and user hits false-positive on a wrong-pairing flag | User can't ship; escape hatch required | Keep parity as warning, not hard block. `--ignore-source-binding-warnings` flag. Document the escape hatch in decision report output. |
| Single-site benchmark doesn't generalize | Ship v0.23 confident, break on site #3 | Benchmark matrix (Dawn-like + Online Store 2.0 + heavily customized + app-heavy) is a success criterion, not a nice-to-have. |

### Completion summary (definition of done)

v0.23 ships when all of:

1. Matrix benchmark shows ≥20% reduction in cross-verify failures AND zero wrong-pairing regressions vs v0.22.
2. Screenshot fallback triggers ≤1 per 5 roles on a typical product-information section.
3. Every source-binding verdict in the decision report includes a readable Liquid file:line citation.
4. `@shopify/liquid-html-parser` dep pinned + audited.
5. Upgrade migration (`/theme-forge upgrade --to v0.23`) is non-destructive (creates `role-bindings.json` if absent, leaves existing anchor maps alone).
6. Regression harness: 3 existing + 4 new + multi-hop snippet chain fixture all pass.
7. PR includes CHANGELOG entry, VERSION bump (0.22.x → 0.23.0), and updated onboard documentation.

### Dismissed alternatives (and why)

- **Confidence-gated review queue only, no source-binding** (Codex's 10x reframing). Skipped: the v0.22 scorer already surfaces tied candidates. Adding a review queue without a stronger signal just relocates the guess to a human. Source-binding adds the signal first; review queue is the screenshot fallback, scoped down.
- **Full runtime instrumentation (inject markers via DOM mutation)** (Codex alt #3). Skipped for v0.23: invasive, requires theme-specific hooks. Valid for v0.24+ if static AST analysis proves insufficient.
- **Ship only the decision-log audit trail, no algorithm change** (Claude subagent's 1-day alt). Reconsidered: the log is part of v0.23 (citations), but shipping it alone leaves the tied-candidate failure mode unsolved. Not sufficient.
- **Regex + state machine Liquid parser** (original plan). Dropped: parser correctness is a v0.23 pillar. Regex on Liquid control flow breaks on nested `{% if %}`, `{%- -%}`, comment tags. Use AST.
